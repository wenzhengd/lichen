# Manuscript Alignment Revision Note

## Status

This note records the current mismatch between:

- the implemented code in `src/lichen`,
- the simplified theory notes in `docs/theory/*`, and
- the more complete construction in `docs/lichen_manuscript.md`.

The manuscript is treated here as the target specification for the next
revision.

## Main Conclusion

The current repository already matches the manuscript on the core blockwise
mathematics:

- one shared hidden Gaussian variable `xi` is reused across the shot,
- toggling-frame generators are built as `A_j = G_j^\dagger (\sum_a Z_a) G_j`,
- projection is delayed until the block boundary,
- exact block probabilities may be computed from Pauli amplitudes
  `q_{b,Q}(xi) = |u_{b,Q}(xi)|^2`,
- the effective block support is restricted to the subgroup generated by the
  block generators.

However, the current code and `docs/theory/*` do not fully implement the
manuscript's simulator-facing frame bookkeeping.

The main gap is:

- the manuscript defines sampled block faults first in the toggling frame as
  `Q_b`,
- then maps them to physical inserted faults
  `F_b = G_{e_b} Q_b G_{e_b}^\dagger`,
- while the current code returns a raw Pauli label per block and the current
  notebook applies that label directly after the ideal block.

This is correct only when the end-of-block Clifford frame leaves the Pauli
label unchanged. In general it is not the manuscript's rule.

## Current State

### What is already aligned

- `src/lichen/frame_tracker.py` correctly builds the cumulative Clifford
  history needed for the toggling frame.
- `src/lichen/segment_generators.py` correctly builds signed Pauli
  generators for each segment.
- `src/lichen/block_probability_search.py` already uses the manuscript's
  central identity `q_{b,Q}(xi) = |u_{b,Q}(xi)|^2`.
- `src/lichen/block_fault_export.py` performs simulator-facing sparsification
  only after exact block probabilities are available.

### What is not yet aligned

- `docs/theory/derivations.md` still presents the block Pauli channel as if it
  were directly the simulator-facing inserted object.
- `src/lichen/block_sampler.py` samples only one Pauli label per block and does
  not convert it from a toggling-frame fault into a physical inserted fault.
- `docs/tutorials/end_to_end_demo.ipynb` applies sampled block labels directly
  after the ideal block, which follows the old convention instead of the
  manuscript's corrected one.

## Target Behavior

The manuscript-aligned simulator should distinguish two different objects for
each block:

1. Toggling-frame sampled block fault `Q_b`.
2. Physical inserted block fault `F_b = G_{e_b} Q_b G_{e_b}^\dagger`.

The block probability calculation remains a toggling-frame calculation.

The simulator-facing sampled circuit should then be interpreted as:

1. apply the ideal block Clifford,
2. insert the physical fault `F_b`,
3. continue with the next block,
4. reuse the same hidden `xi` for all blocks in the shot.

This should be the only accepted interpretation in docs, tests, and notebook
examples after the revision.

## Required Code Revisions

### 1. Separate toggling-frame and physical sampled faults

Revise the block-sampling data model so that it can carry both:

- the sampled toggling-frame label `Q_b`,
- the mapped physical inserted label `F_b`.

Minimal direction:

- keep the current block probability distribution as the toggling-frame
  conditional distribution,
- add explicit end-of-block frame conjugation using `G_{e_b}`,
- expose both labels in the sampled process object.

### 2. Add block-end frame mapping

The sampler should compute the cumulative Clifford at the end of each block and
map each nontrivial sampled toggling-frame Pauli into the physical frame before
returning the simulator-facing result.

### 3. Keep probability construction unchanged unless needed

The block probability engine appears conceptually aligned already. The main
revision target is not the probability formula itself, but the meaning of the
sampled output object.

### 4. Revisit naming

Names like `block_faults` are currently ambiguous. The revised API should make
the frame explicit. Example directions:

- `toggling_frame_fault`,
- `physical_fault`,
- `toggling_frame_probabilities`.

Exact names can be chosen later, but the ambiguity should be removed.

## Required Documentation Revisions

These documentation changes should be made after the code behavior is updated.

### 1. Replace the simplified process formula in `docs/theory/*`

`docs/theory/derivations.md` should stop presenting the block Pauli channel as
the final inserted object. It should instead:

- define the toggling-frame block export first,
- then state the mapping to physical inserted faults,
- then explain the simulator-facing sampled circuit in physical-frame terms.

### 2. Update the tutorial notebook

The tutorial should stop applying sampled block labels directly after each ideal
block unless they have first been conjugated into the physical frame.

### 3. Keep the manuscript and lightweight theory notes consistent

After revision, `docs/theory/*` should become a concise presentation of the
same story already written in `docs/lichen_manuscript.md`, not a simplified
alternative convention.

## Acceptance Checks

The revision should be considered complete only if the following are true.

### Behavioral checks

- Free vs DD behavior is still preserved at `window_size = 2`.
- A circuit whose block ends in a nontrivial Clifford frame shows that
  toggling-frame `Q_b` and physical `F_b` can differ.
- The notebook's state-propagation helper uses physical inserted faults, not
  raw toggling-frame labels.

### API checks

- The sampled process object clearly distinguishes toggling-frame and physical
  faults.
- Public docstrings state which frame each returned Pauli label belongs to.

### Documentation checks

- `docs/theory/*` matches the revised code behavior.
- The tutorial examples follow the same frame convention as the code.

## Non-Goals For This Revision

This revision note does not propose a broader physics change to the hidden
memory model itself. In particular, it does not yet change:

- the shared quasi-static scalar model,
- the Clifford-only control restriction,
- the fixed-window partition rule,
- or the current exact sparse block-probability machinery.

Those may be revised later, but they are separate from the immediate manuscript
alignment issue.

## Recommended Order

1. Approve this manuscript-aligned target behavior.
2. Update the sampler and sampled-process API to carry both frame views.
3. Add tests for toggling-frame versus physical inserted faults.
4. Update the notebook helpers.
5. Rewrite `docs/theory/*` to match the corrected implementation.

This order avoids rewriting public documentation around behavior the code does
not yet implement.
